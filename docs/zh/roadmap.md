### 与成熟编排平台（如 Kubernetes）的差距分析

以 Kubernetes 为参照，从以下维度评估差距：

#### 1. 分布式与高可用（差距：极大）

| 能力 | K8s | ClawFactory 现状 |
|------|-----|-----------------|
| 多节点部署 | ✅ 集群模式 | ❌ 单进程单机 |
| 控制平面高可用 | ✅ etcd 集群 + leader 选举 | ❌ 单点故障 |
| 数据存储高可用 | ✅ etcd 分布式一致性 | ❌ SQLite 单文件 |
| 水平扩展 | ✅ 自动扩缩容 | ❌ 不支持 |
| 服务发现 | ✅ DNS + Service | ❌ 仅本地注册 |

#### 2. 调度能力（差距：大）

| 能力 | K8s | ClawFactory 现状 |
|------|-----|-----------------|
| 多维度调度 | ✅ 资源、亲和性、污点容忍 | ⚠️ 仅能力标签匹配 |
| 负载均衡 | ✅ 多种策略 | ✅ 基于最少活跃任务数的负载均衡（v0.2） |
| 抢占调度 | ✅ 优先级抢占 | ❌ 不支持 |
| 资源配额 | ✅ ResourceQuota | ❌ 不支持 |
| 调度队列 | ✅ 多队列 | ⚠️ 单队列 |

#### 3. 工作流能力（差距：中等）

| 能力 | K8s (Argo Workflows) | ClawFactory 现状 |
|------|---------------------|-----------------|
| DAG 工作流 | ✅ | ✅ 基础实现 |
| 条件分支 | ✅ | ❌ 不支持 |
| 循环/重试 | ✅ 灵活配置 | ⚠️ 仅固定次数重试 |
| 参数传递 | ✅ 模板参数 | ⚠️ 仅 input/output 键值对 |
| 超时控制 | ✅ | ❌ 不支持 |
| 暂停/恢复 | ✅ | ❌ 不支持 |
| 工作流模板 | ✅ | ❌ 不支持 |
| 子工作流 | ✅ | ❌ 不支持 |
| Cron 调度 | ✅ | ❌ 不支持 |

#### 4. 可观测性（差距：中等）

| 能力 | K8s | ClawFactory 现状 |
|------|-----|-----------------|
| 结构化日志 | ✅ | ✅ slog JSON 格式 + 日志级别控制（v0.3） |
| 指标监控 | ✅ Prometheus | ✅ Prometheus `/metrics` 端点 + 7 个自定义指标（v0.3） |

#### 5. 安全性（差距：大）

## 技术演进方向总结

ClawFactory 的技术演进遵循以下核心原则：

1. **平台优先于框架**：作为独立编排服务运行，通过标准协议（ARI）接入智能体，而非嵌入应用进程。这决定了我们优先强化控制平面能力和协议标准化。

2. **渐进式复杂度**：从静态 DAG → 条件分支 → 运行时修改 → LLM 驱动规划，每一步都建立在前一步的可靠基础上。不跳跃式引入复杂度。

3. **接口抽象保障演进**：StateStore、TaskQueue、SharedMemory 等核心接口的抽象设计，确保存储后端、消息层、记忆机制可以独立演进而不影响上层逻辑。

4. **安全内建而非后补**：从 v0.1 的静态 Token + RBAC，到 v0.4 的 JWT + TLS，再到远期的声明式策略语言和信任评分，安全能力随平台成熟度同步提升。

5. **环境适应性**：SQLite 单机开发 → PostgreSQL 私有化 → K8s 云原生 → 多集群联邦 → 边缘计算，同一套代码适配不同部署场景。

详细的技术演进方向参见 `docs/zh/architecture.md` 的"技术演进方向"章节。
